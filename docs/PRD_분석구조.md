# PRD: 2단계 분석 구조

## 개요

사용자가 포트폴리오 문서를 업로드하면 **2단계**로 나누어 분석합니다.
1단계에서 문서 키워드를 추출하고, 2단계에서 키워드 기반으로 합격 포트폴리오를 매칭하여 비교 분석합니다.

---

## UX 흐름

```
파일 업로드
  ↓
(크레딧 유저) 차감 확인 모달 → 확인
  ↓
[1단계: 키워드 스캔]
  - 파일에서 텍스트 추출 (PDF → extractTextFromPdf)
  - Claude API 경량 호출 → 키워드 5~10개 추출
  - 로딩 화면: "문서를 스캔하는 중..."
  ↓
키워드 편집 모달
  - AI 추출 키워드를 칩(chip) 형태로 표시
  - 사용자가 키워드 추가/삭제 가능
  - "합격작 비교 시작" 버튼
  ↓
[2단계: 합격작 비교]
  - 확정 키워드로 portfolios.tags 매칭
  - 매칭된 합격작을 셔플 (매번 다른 결과)
  - Claude API 전체 분석 (15개 카테고리 + 가독성 + 레이아웃)
  - 로딩 화면: 기존 로딩 메시지들
  ↓
결과 표시
```

---

## 1단계: 키워드 스캔

### 서버 액션
- **함수**: `extractKeywords(extractedText, fileName)`
- **위치**: `app/actions/analyze.ts`
- **모델**: Claude Sonnet (경량)
- **입력**: 문서 텍스트 앞 3000자 + 파일명
- **출력**: JSON 배열 `["시스템기획", "강화", "UI/UX", ...]`
- **비용**: 기존 분석의 1/10 이하 (짧은 프롬프트)

### 폴백
텍스트 추출 실패 또는 텍스트 부족 시 → 파일명에서 키워드 패턴 매칭

---

## 2단계: 합격작 비교 분석

### DB 매칭 로직
1. `portfolios` 테이블 전체 로드 (187개, 작은 데이터셋)
2. 각 포트폴리오의 `tags text[]`와 사용자 키워드 부분 일치 매칭
   - `tag.includes(keyword) || keyword.includes(tag)`
3. 매칭된 결과를 **Fisher-Yates 셔플**
4. 셔플된 상위 30개를 분석 컨텍스트로 사용
5. 매칭 5개 미만 → 전체 셔플 폴백

### 셔플 메커니즘
- **목적**: 동일 키워드로 재분석해도 매번 다른 합격작 참조
- **구현**: Fisher-Yates 알고리즘 (균일 분포 보장)
- **효과**: "앞 문서만 보고 매번 같은 대답" 문제 해결

### 시스템 프롬프트
키워드 컨텍스트가 프롬프트에 주입됨:
```
## 사용자 확정 키워드: 시스템기획, 강화, 재화
이 키워드와 관련된 합격 포트폴리오 25개를 기반으로 평가합니다.
```

---

## 크레딧 정책

- 1단계(키워드 추출) + 2단계(합격작 비교) = **합산 1크레딧**
- 키워드 추출은 경량 호출이므로 추가 과금 없음
- 크레딧 차감은 기존 로직 유지 (analyzeDocumentDirect 내부)

## 버튼/단계 명칭

| 단계 | 명칭 | 설명 |
|------|------|------|
| 1단계 로딩 | "1단계: 키워드 스캔" | 자동 실행 |
| 1단계 완료 | "키워드 스캔 완료" | 편집 가능 |

## 수정 파일

| 파일 | 변경 |
|------|------|
| `app/actions/analyze.ts` | `extractKeywords()` 추가, `analyzeDocumentDirect()` keywords 파라미터 + 태그 매칭 + 셔플 |
| `components/analyze-dashboard.tsx` | 카테고리 선택 모달 → 키워드 편집 UI, 2단계 흐름 |
| `lib/document-categories.ts` | 삭제 (고정 카테고리 → 동적 키워드로 전환) |

## 향후 개선 가능

- `portfolio_analysis.key_features`도 매칭 대상에 추가 (현재는 `portfolios.tags`만)
- 키워드 자동완성 (DB 태그 목록 기반)
- 매칭 통계 표시 ("25개 합격작과 비교합니다")
