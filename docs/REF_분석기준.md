| 2 | **구체성** | 구체성 | 수치, 데이터, KPI 포함 여부. "좋다", "많다" 같은 모호한 표현은 감점. |

| 5 | **창의성** | 창의성 | 독창적인 아이디어, 차별화 요소. 일반적인 내용만 있으면 감점. |

### 게임 디자인 역량 (10개)

| # | 항목명 | subject 키 | 설명 |
|---|--------|-----------|------|
| 6 | **핵심 반복 구조** | 핵심반복구조 | 플레이어가 가장 자주 반복하는 행동의 흐름(예: 이동→전투→보상→강화→다시 이동)이 명확하게 정의되어 있는가. 버튼을 누르면 무슨 일이 일어나고, 그 결과가 플레이어에게 어떻게 전달되는지 구체적인가. |
| 7 | **콘텐츠 분류 체계** | 콘텐츠분류 | 게임의 주요 요소들이 겹치지 않고 빠짐없이 나뉘어 정리되었는가. 큰 분류에서 작은 분류로 체계적으로 나누는 구조가 있는가. |
| 8 | **재화 흐름 설계** | 재화흐름 | 게임 안에서 돈이나 자원이 어디서 생기고, 어디에 쓰이고, 어디서 사라지는지가 설계되었는가. 자원이 한없이 쌓이기만 하는 문제를 막는 구조가 있는가. |
| 9 | **플레이 경험 목표** | 플레이경험 | 플레이어가 느꼈으면 하는 감정이나 경험(도전감, 탐험의 재미, 이야기 몰입 등)이 명확하고, 그 경험을 만들기 위해 어떤 게임 규칙을 넣었는지 연결이 되는가. |
| 10 | **수치 데이터 정리** | 수치데이터 | 게임에 들어가는 숫자 데이터가 표 형태로 정리되어 있는가. 문서 주제에 해당하는 수치가 있는가. 예시 데이터가 포함되어 있고, 각 표끼리 어떻게 연결되는지 설명이 있는가. |
| 11 | **기능 간 연결 관계** | 기능연결 | 핵심 기능이 무엇이고, 추가 기능들(아이템, 스킬, 장비 등)이 어떤 순서로 연결되는지 그림이나 도표로 표현되었는가. 우선순위가 있는가. |
| 12 | **동기 부여 설계** | 동기부여 | 게임을 왜 계속하고 싶게 만드는지가 설계되었는가. 단기/중기/장기 목표가 단계별로 나뉘어 있는가. |
| 13 | **난이도 균형** | 난이도균형 | 초반~후반 난이도 곡선, 잘하는 사람이 너무 강해지는 걸 막는 장치가 있는지 고려되었는가. |
| 14 | **화면 및 조작 설계** | 화면조작 | 게임 화면 구성(메뉴, 버튼 배치 등)의 밑그림이 포함되었는가. 조작 방법이 통일되어 있으면서도 다양한 경험을 제공하는가. |
| 15 | **개발 일정 및 산출물** | 개발일정 | 개발 단계 계획(기획→첫 버전→테스트→출시)이 있는가. 기능 목록, 테스트 항목, 수치 조정표 같은 실무 문서가 포함되었는가. |

### 채점 규칙

- **0~100점** (엄격하게)
- 게임 기획서가 아닌 일반 포트폴리오인 경우: 관련 내용이 전혀 없으면 0점, 간접 언급이 있으면 그 수준에 맞게
- 문서의 주제에 맞게 평가 (캐릭터 문서에 "몬스터 데이터 없다" 같은 피드백 금지)

### 점수 기준

| 점수 | 등급 | 의미 |
|------|------|------|
| 90~100 | S | 즉시 합격 수준 |
| 80~89 | A | 합격 가능 수준 |
| 70~79 | B | 보완 필요 |
| 60~69 | C | 상당한 보완 필요 |
| 0~59 | D | 전면 재작성 권장 |

## 4. 추가 평가 요소

### 강점/보완점
- 강점 6개, 보완점 6개 필수
- 각각 서로 다른 관점
- 실제 문서에서 확인된 내용 기반

### 회사별 비교 피드백
- 넥슨, 엔씨소프트, 넷마블, 크래프톤, 스마일게이트, 펄어비스, 네오위즈, 웹젠 (8개)
- 각 2~3문장, 합격자 특징 vs 이 문서 비교
- **벤치마크 데이터 기반**: `data/company-benchmarks.json`의 회사별 20개 항목(design 10 + readability 10) 데이터를 프롬프트에 주입하여 비교 근거 강화

### 평가 방법 (합격 사례 비교)
1. **문서 구조**: 합격 문서들은 체계적 구조(개요→분석→설계→검증)를 가짐
2. **수치/데이터**: 합격 문서들은 구체적 KPI, 수치 목표가 있음
3. **시각 자료**: 합격 문서들은 다이어그램, 플로우차트, 표를 적극 활용
4. **비교 분석**: 합격 문서들은 레퍼런스 분석, 경쟁 타이틀 비교가 있음
5. **기술적 깊이**: 합격 문서들은 기술 구현 방안, 제약사항을 다룸

## 5. 채점 주의사항

1. **거짓 칭찬 금지**: 없는 것을 있다고 하지 마세요
2. **강점/보완점 모순 금지**: 같은 주제를 양쪽에 넣지 않음
3. **합격 사례 번호 언급 금지**: "합격 사례 1번" 같은 표현 불가
4. **후한 점수 금지**: 대부분의 문서는 60~80점대
5. **주제 맞춤 평가**: 문서 주제와 무관한 피드백 넣지 않음

## 6. 회사별 벤치마크 데이터 연동 (✅ 구현 완료)

> 데이터 파일: `data/company-benchmarks.json`
> 데이터 출처: NotebookLM에 187개 합격 포트폴리오를 업로드 → 회사별/항목별 특징 추출
> 상태: **구현 완료 — 테스트 필요**

### 서비스 동작 방식

사용자가 문서를 올리면:

1. **8개사 비교 피드백(companyFeedback)**: 넥슨, 네오위즈, 넷마블, 엔씨소프트, 크래프톤, 펄어비스, 스마일게이트, 웹젠 — **8개 회사 전부에 대해 각각 비교**하여 "이 회사 합격자들은 이렇게 하는데 이 문서는 이렇다"는 피드백 생성
2. **20개 항목 벤치마크 평가**: 게임 디자인 역량 10개 + 문서 가독성 10개 항목의 feedback을 작성할 때, 해당 항목의 벤치마크 데이터를 참고하여 합격자 기준 대비 구체적 비교 제공

### 일반회사 데이터의 역할

> **일반회사는 사용자에게 별도로 노출되지 않는다.** 일반회사 데이터는 8개 회사 각각의 벤치마크 데이터에 **섞여서** Claude의 판단 기준을 다각화하는 용도로만 사용된다.

- 8개 회사 벤치마크: 해당 회사에 특화된 합격자 특징
- 일반회사 벤치마크: 특정 회사에 국한되지 않는 업계 공통 합격 기준
- Claude API 프롬프트에 8개 회사 데이터 + 일반회사 데이터를 **함께 주입** → Claude가 회사별 고유 특징과 업계 공통 기준을 모두 고려하여 판단
- 이를 통해 특정 회사 데이터만으로는 놓칠 수 있는 관점을 보완하고, 피드백의 다각화를 달성

### 데이터 구조

```json
{
  "회사명": {
    "design": {
      "핵심반복구조": "이 회사 합격자들의 핵심반복구조 특징 설명...",
      "콘텐츠분류": "...",
      // ... 10개 항목
    },
    "readability": {
      "글자크기구분": "이 회사 합격자들의 글자크기구분 특징 설명...",
      "문단나누기": "...",
      // ... 10개 항목
    }
  }
}
```

### Claude API 호출 설정

| 분석 유형 | 함수명 | max_tokens | 벤치마크 범위 | 비고 |
|-----------|--------|-----------|-------------|------|
| PDF/문서 분석 | `analyzeDocumentDirect` | **16384** | design + readability | 15카테고리 + 10가독성 + 3레이아웃 + 8회사피드백 |
| URL 분석 | `analyzeUrlDirect` | 8192 | design만 | 가독성/레이아웃 평가 없어 출력량 적음 |

- `stop_reason === "max_tokens"` 체크로 응답 잘림 감지 (양쪽 함수)
- 벤치마크 항목당 최대 150자 truncation 적용 (`BENCHMARK_MAX_CHARS`)
- PDF 분석이 URL보다 출력량이 2배 이상 많아 max_tokens 차등 설정
